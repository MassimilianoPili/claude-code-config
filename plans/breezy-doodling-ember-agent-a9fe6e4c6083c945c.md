## 1. A2A Core Components (from spec v1.0)

The A2A protocol is built on **3 layers**:

1. **Canonical Data Model** (Layer 1): Core data structures -- Task, Message, AgentCard, Part, Artifact, Extension
2. **Abstract Operations** (Layer 2): Send Message, Stream Message, Get Task, List Tasks, Cancel Task, Get Agent Card
3. **Protocol Bindings** (Layer 3): JSON-RPC 2.0, gRPC, HTTP+JSON/REST (all three are officially supported)

### Core Concepts

- **A2A Client**: initiates requests to an A2A Server
- **A2A Server (Remote Agent)**: exposes an A2A-compliant endpoint
- **Agent Card**: JSON metadata document describing identity, capabilities, skills, endpoint, auth
- **Message**: a communication turn (role: "user" or "agent"), containing one or more Parts
- **Task**: stateful unit of work with lifecycle (submitted -> working -> completed/failed/canceled/rejected)
- **Part**: smallest content unit -- text, file (raw bytes or URL), or structured data (JSON)
- **Artifact**: output generated by the agent as task result, composed of Parts
- **Context**: optional identifier grouping related tasks/messages

### Task States (TaskState enum)

```
TASK_STATE_UNSPECIFIED   = 0
TASK_STATE_SUBMITTED     = 1  (acknowledged)
TASK_STATE_WORKING       = 2  (actively processing)
TASK_STATE_COMPLETED     = 3  (terminal -- success)
TASK_STATE_FAILED        = 4  (terminal -- error)
TASK_STATE_CANCELED      = 5  (terminal)
TASK_STATE_INPUT_REQUIRED = 6 (interrupted -- needs user input)
TASK_STATE_REJECTED      = 7  (terminal -- agent declined)
TASK_STATE_AUTH_REQUIRED  = 8  (interrupted -- needs authentication)
```

### Task Lifecycle

```
Client sends Message --> Server creates Task (SUBMITTED)
                    --> Server processes (WORKING)
                    --> Server may request input (INPUT_REQUIRED)
                    --> Client sends follow-up Message with same taskId
                    --> Server completes (COMPLETED) or fails (FAILED)
```

Three update delivery mechanisms:
1. **Polling**: GET /tasks/{id} periodically
2. **Streaming**: SSE via POST /message:stream or POST /tasks/{id}:subscribe
3. **Push Notifications**: webhooks -- HTTP POST to client-registered endpoints

## 2. Agent Card Schema (`.well-known/agent-card.json`)

### Discovery Mechanism

Standard path: `https://{domain}/.well-known/agent-card.json` (per RFC 8615)

Three discovery strategies:
1. **Well-Known URI** -- recommended for public agents
2. **Curated Registries** -- enterprise/marketplace environments
3. **Direct Configuration** -- hardcoded or config-file based

### AgentCard Proto Definition (normative source: `specification/a2a.proto`)

```protobuf
message AgentCard {
  string name = 1;                                    // REQUIRED
  string description = 2;                             // REQUIRED
  repeated AgentInterface supported_interfaces = 3;   // REQUIRED -- ordered, first is preferred
  AgentProvider provider = 4;                         // optional
  string version = 5;                                 // REQUIRED
  optional string documentation_url = 6;
  AgentCapabilities capabilities = 7;                 // REQUIRED
  map<string, SecurityScheme> security_schemes = 8;
  repeated SecurityRequirement security_requirements = 9;
  repeated string default_input_modes = 10;           // REQUIRED -- media types
  repeated string default_output_modes = 11;          // REQUIRED -- media types
  repeated AgentSkill skills = 12;                    // REQUIRED
  repeated AgentCardSignature signatures = 13;        // optional JWS signatures
  optional string icon_url = 14;
}
```

### Supporting Types

```protobuf
message AgentInterface {
  string url = 1;               // REQUIRED -- absolute HTTPS URL
  string protocol_binding = 2;  // REQUIRED -- "JSONRPC", "GRPC", "HTTP+JSON"
  string tenant = 3;            // optional
  string protocol_version = 4;  // REQUIRED -- e.g. "1.0"
}

message AgentProvider {
  string url = 1;           // REQUIRED
  string organization = 2;  // REQUIRED
}

message AgentCapabilities {
  optional bool streaming = 1;
  optional bool push_notifications = 2;
  repeated AgentExtension extensions = 3;
  optional bool extended_agent_card = 4;
}

message AgentSkill {
  string id = 1;            // REQUIRED
  string name = 2;          // REQUIRED
  string description = 3;   // REQUIRED
  repeated string tags = 4; // REQUIRED
  repeated string examples = 5;
  repeated string input_modes = 6;   // overrides agent defaults
  repeated string output_modes = 7;  // overrides agent defaults
  repeated SecurityRequirement security_requirements = 8;
}
```

## 3. A2A vs MCP: Complementary, NOT Competing

This is explicitly addressed in the A2A documentation and DeepLearning.AI course:

### MCP (Model Context Protocol)
- **Purpose**: Connect an LLM to **tools and data sources** (databases, APIs, file systems)
- **Relationship**: Client (LLM) <-> Server (tool provider)
- **Metaphor**: "Giving an agent hands" -- extending what a single agent can do
- **Visibility**: The client sees the server's internal tools, resources, prompts
- **Transport**: stdio, SSE, Streamable HTTP

### A2A (Agent-to-Agent Protocol)
- **Purpose**: Enable **opaque agents** to communicate and collaborate
- **Relationship**: Agent <-> Agent (peer-to-peer collaboration)
- **Metaphor**: "Letting agents talk to each other" -- agents as black boxes
- **Visibility**: Agents do NOT expose internal state, memory, or tools to each other
- **Transport**: JSON-RPC 2.0, gRPC, HTTP+JSON/REST

### Key Distinction: Opacity

MCP intentionally exposes tools/resources. A2A intentionally preserves opacity.

In A2A, agents communicate through **messages** and **tasks**, not by directly invoking each other's tools. An A2A server receives a natural-language (or structured) message and returns task results -- it never exposes its internal MCP tools to the calling agent.

### How They Work Together

```
[Agent A] --A2A--> [Agent B's A2A endpoint]
                        |
                        v
                   [Agent B's LLM]
                        |
                        v
                   [Agent B's MCP tools]
                   (invisible to Agent A)
```

A2A wraps MCP. Your scenario (exposing ~290 MCP tools via an A2A endpoint) is the canonical use case.

## 5. Required HTTP Endpoints for A2A-Compliant Server (HTTP+JSON/REST binding)

### Minimum Required Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/message:send` | Send message (create/continue task) |
| `POST` | `/message:stream` | Send message with SSE streaming |
| `GET` | `/tasks/{id}` | Get task status |
| `GET` | `/tasks` | List tasks (with query params) |
| `POST` | `/tasks/{id}:cancel` | Cancel task |

### Optional Endpoints (if capabilities declared)

| Method | Endpoint | Capability Required |
|--------|----------|-------------------|
| `POST` | `/tasks/{id}:subscribe` | `streaming: true` |
| `POST` | `/tasks/{id}/pushNotificationConfigs` | `pushNotifications: true` |
| `GET` | `/tasks/{id}/pushNotificationConfigs/{configId}` | `pushNotifications: true` |
| `GET` | `/tasks/{id}/pushNotificationConfigs` | `pushNotifications: true` |
| `DELETE` | `/tasks/{id}/pushNotificationConfigs/{configId}` | `pushNotifications: true` |
| `GET` | `/extendedAgentCard` | `extendedAgentCard: true` |

### Discovery Endpoint (separate from the A2A service)

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/.well-known/agent-card.json` | Agent Card discovery (RFC 8615) |

### Protocol Requirements
- **Content-Type**: `application/json` for requests/responses
- **Streaming**: SSE (`text/event-stream`) for stream endpoints
- **Versioning**: `A2A-Version: 1.0` header on every request
- **JSON naming**: camelCase (not snake_case from proto)
- **Enum values**: SCREAMING_SNAKE_CASE strings (e.g., `"TASK_STATE_COMPLETED"`)
- **Timestamps**: ISO 8601 UTC (`"2025-10-28T10:30:00.000Z"`)
- **Multi-tenant**: all endpoints support optional `/{tenant}/` prefix

## 6. Implementation Plan for SOL

### Architecture: A2A Gateway wrapping existing MCP Server

```
Internet --> nginx (:8888)
               |
               +--> /.well-known/agent-card.json  (static file or Go handler)
               |
               +--> /mcp/a2a/*  (A2A Gateway -- new Go service)
                        |
                        v
                   [A2A Gateway]  -- translates A2A messages to MCP tool calls
                        |
                        v
                   [mcp-proxy :8098]  -- existing MCP SSE endpoint
                        |
                        v
                   [simoge-mcp :8099]  -- existing ~290 tools
```

### What the A2A Gateway Needs to Do

1. **Serve Agent Card** at `/.well-known/agent-card.json` (or a custom path)
2. **Accept A2A messages** at `/mcp/a2a/v1/message:send` (and `:stream`)
3. **Translate** incoming A2A messages into MCP tool calls:
   - Parse the user's natural-language or structured message
   - Either use an LLM to select the right MCP tool, or accept structured `data` Parts that directly specify tool name + arguments
   - Call the MCP server via its Streamable HTTP transport
4. **Manage task state** (in-memory or Redis): task creation, status transitions, history
5. **Return artifacts** as A2A Parts (text, JSON data, files)
6. **Auth**: reuse existing HMAC API key scheme from mcp-proxy

### Option A: Thin Gateway (no LLM, structured input only)

The A2A client sends a message with a structured `data` Part:
```json
{
  "message": {
    "role": "ROLE_USER",
    "parts": [{
      "data": {
        "tool": "docker_list_containers",
        "arguments": {"status": "running"}
      },
      "mediaType": "application/json"
    }]
  }
}
```

The gateway maps this directly to an MCP `tools/call` request. No LLM needed.
Pro: Simple, deterministic. Con: Not truly "agentic" -- the caller must know tool names.

### Option B: LLM-Backed Gateway (natural language)

The A2A client sends a natural-language message:
```json
{
  "message": {
    "role": "ROLE_USER",
    "parts": [{"text": "Show me all running Docker containers"}]
  }
}
```

The gateway uses Ollama (local LLM) to:
1. Interpret the request
2. Select the appropriate MCP tool(s)
3. Execute and format the response

Pro: True agent behavior. Con: Requires LLM inference, more complex.

### Option C: Hybrid (recommended)

Support both modes:
- If message contains a `data` Part with `tool` field -> direct MCP dispatch
- If message contains only `text` Parts -> route through LLM for tool selection
- Expose both as different **skills** in the Agent Card

### Technology Choice

Given the existing Go infrastructure (mcp-proxy, proxy-ai, knowledge-graph), **Go** is the natural choice.
The A2A Go SDK (`github.com/a2aproject/a2a-go`) provides ready-made types and handlers.

### Deployment

- New directory: `/data/massimiliano/a2a-gateway/`
- Docker container on `shared` network (like all other services)
- Nginx route: `/mcp/a2a/` -> a2a-gateway container
- Agent Card: served at `sol.massimilianopili.com/.well-known/agent-card.json` (nginx static file) AND at `sol.massimilianopili.com/mcp/a2a/v1/extendedAgentCard` (authenticated)

### Skills Mapping (subset of ~290 MCP tools -> A2A skills)

Rather than exposing all 290 tools as individual A2A skills (which would be unwieldy), group them into logical skill categories:

| A2A Skill ID | MCP Tools Covered | Description |
|-------------|-------------------|-------------|
| `infrastructure` | docker_*, ocp_* | Container and cluster management |
| `database` | db_*, graph_* | SQL queries, graph operations |
| `research` | web_search, web_fetch, openalex_* | Academic research and web search |
| `devops` | gitea_*, devops_*, jira_* | Git, CI/CD, project management |
| `monitoring` | docker_get_container_stats, recovery_* | Health checks and monitoring |
| `knowledge` | embeddings_*, anki_* | Knowledge graph and flashcards |
| `files` | fs_*, s3_*, pdf_* | File operations and storage |
| `auth` | keycloak_*, auth_* | Identity and access management |

## Serendipitous Connections

### A2A <-> Agent Framework project
This is directly relevant to the **Agent Framework** project (`/data/massimiliano/agent-framework/`). The Agent Framework does multi-agent orchestration -- A2A provides the standardized protocol for inter-agent communication that the framework currently lacks. The A2A Java SDK (`a2aproject/a2a-java`) could be integrated into the Spring Boot agent framework.

### A2A <-> SocialMCP project
The **SocialMCP** project (autopoietic knowledge network) envisions federated agent communication. A2A's Agent Card discovery mechanism (`.well-known/agent-card.json`) is exactly the kind of decentralized discovery that SocialMCP needs. Each node in the network could advertise its capabilities via A2A Agent Cards.

### A2A Agent Cards <-> DNS/PKI
The Agent Card signing mechanism (JWS with JWKS endpoint) mirrors the web PKI trust model. The `AgentCardSignature` field allows cryptographic verification of agent identity -- relevant to the **DSS Wrapper** project's digital signature work.
